# Default Pack

## Planning Bias

- Start with broad product clarification, then narrow into architecture and stack only where needed.
- Prefer ambitious but still testable product scope.

## Implementation Bias

- Choose the smallest reasonable stack that satisfies the locked decisions.
- Prefer clear, maintainable boundaries over clever abstraction.

## Evaluation Bias

- Be skeptical of thin slices, placeholders, and file-only claims.
- Require evidence from the running product whenever the contract expects user-facing behavior.
